# Spec-First: 系统性思维启动器

> 写代码之前，先把问题和方案想清楚。

**核心理念：**

不急着给答案，先搞清楚要解决什么问题。代码是最后一步，不是第一步。

---

## 触发方式

说清楚你的需求就行，例如：

- "帮我做个用户注册流程"
- "我想做一个邮件分类系统"
- "/think 怎么设计一个待办清单"

不需要说"用 spec-first"，我会自动判断并引导。

---

## 完整流程

### Stage 0: Triage（分流门）

**目标：** 在进入 7 阶段之前先判断流程重量——不是所有任务都需要走完整流程。

**3 个判断问题（每个 Y/N）：**

| # | 问题 | Y 表示 |
|---|------|-------|
| 1 | 需求是否只有一种合理理解？ | 无歧义 |
| 2 | 方案是否只有 1-2 种合理选择？ | 选择空间窄 |
| 3 | 失败成本是否低？ | 出错可快速纠正 |

**时间预算（根据复杂度自动估算）：**

| 结果 | 路径 | 时间预算 |
|------|------|----------|
| **3Y** | 跳过 Stage 1-4，直接 Execute | **≤ 5 分钟** |
| **2Y** | 跳过 Stage 1-2（Problem Exploration + Solution Options），从 Design 开始 | **≤ 15 分钟** |
| **0-1Y** | 走完整 7 阶段流程 | **30-60 分钟** |

**为什么要这个门**：
- 7 阶段对简单任务太重（昨晚 review + 今晚 challenge 独立验证）
- Planning Fallacy (Kahneman & Tversky, 1979) 显示人倾向低估任务复杂度——Triage 提供早期 complexity 信号
- 跨 skill 共享的 spec-first/LOOP.md ux-iteration 维度多次记录"流程太重"反馈

**何时跳过 Triage**：用户明确说"不用问，直接做完整流程"——但仍要走完 Stage 1-7。

---

### Stage 1: Clarify — 澄清问题

**目标：** 搞清楚到底要解决什么问题，以及为什么解决这个问题。

**两个子阶段：**

#### 1.1 Problem Exploration（问题探索）

**时机：** 用户需求含糊、不熟悉、或涉及多人协作时触发。简单明确的需求可跳过。

**判断标准：** 如果用户需求满足以下任一，跳过此阶段：
- 需求非常明确（"帮我把A文件的第3行改成X"）
- 纯执行任务（"帮我运行这个脚本"）
- 用户明确说"不用问，直接做"

**我会：** 深度优先追问——沿一条线索打到底，再换下一个分支：

| 问题类型 | 适用场景 | 目的 |
|----------|----------|------|
| "具体是什么场景？" | 需求含糊 | 理解真实使用情况 |
| "之前试过什么？结果如何？" | 复杂问题 | 避免重复、借鉴已有 |
| "有什么限制？" | 新方向 | 明确约束 |
| "成功是什么样？" | 所有问题 | 明确目标 |
| "最坏情况会怎样？" | 重要决策 | 风险意识 |

**追问规则（Grill-Me 齿轮）：**

1. **每次只问一个问题** — 不要批量抛出多个问题，形成问→答→追问→答的深度对话链
2. **每个问题必给推荐答案** — 基于已有上下文先给猜测/建议，用户只需确认或修正
3. **能查就别问** — 如果问题能通过读文件、查代码、搜索上下文回答，AI 自己查，不问用户；只有需要用户主观判断/业务上下文/偏好决策的问题才问
4. **深度优先** — 沿当前线索追问到清晰为止，再换下一个分支；不要浅尝辄止就跳方案
5. **直到共识才停** — 终止条件不是"问完 N 个"，而是双方理解对齐

**追问示例：**
```
用户：我想做一个邮件分类系统

我（在进入方案前）："具体是什么场景？我猜测是每天上班前
         批量处理新邮件，而不是整理历史归档——对吗？"
         （推荐答案 + 单问）

用户：对，就是每天早上的新邮件

我："之前试过什么方法？我看你有 email_organizer.py，
         猜你之前用过规则过滤但不够用？"
         （能查就先查 + 推荐答案）

用户：对，规则加不过来

我："成功是什么样？我建议定义为：早上8点前收到摘要，
         5分钟内处理完重要邮件。符合你的预期吗？"
         （推荐答案 + 单问）

用户：对

[共识检查后，进入1.2]
```

**原则：**
- 问具体，不问抽象
- 用"比如"引导举例，不要问"能不能"之类的是非题
- 追问到双方理解对齐即可——简单需求可能 2 问就够，复杂需求可以追问 5-10 轮
- **共识检查**：追问结束时，AI 主动用一句话总结理解，等用户确认后再进入 1.2：
  ```
  [共识检查] 我目前的理解：
  1. 问题是…（一句话）
  2. 核心约束是…
  3. 成功标准是…
  如果准确，我进入方案设计。需要修正吗？
  ```

#### 1.1.bis Reference Class 历史检索（可选子阶段）

**何时触发**：当"之前试过什么？结果如何？" 的答案不充分，或项目有多个相似历史项目时，主动触发。

**目的**：对冲 Planning Fallacy（Kahneman & Tversky, 1979）—— 不只依赖当前项目细节（"inside view"），还参考历史相似项目的统计分布（"outside view"）。

**我会：**
1. 问"过去有没有做过类似的事？最终时间/效果如何？"
2. 如果有历史数据：列出参考点（"上次 X 项目花了 Y，最终效果 Z"）
3. 用历史校准当前预估："基于类似项目，这任务预计时间范围是 A-B"

**何时跳过**：
- 完全陌生的领域（无历史参考）
- 用户明确说"不需要参考历史"
- 任务极小（已在 Stage 0 标记 3Y）

#### 1.2 Solution Options（方案选项）

**前提：** 问题探索完成（或跳过），进入方案选项。

**执行方式：** 用 `question` 工具直接给出选项，不要先问开放性问题。

**我会：**
- 一次性给出 2-4 个互斥选项
- 选项要具体、可操作
- 每选项附简短说明

**示例（反例 vs 正例）：**

❌ 反例（先问开放问题）：
```
你希望达到什么效果？
请描述一下你的需求...
```

✅ 正例（直接给选项）：
```
基于你的需求，我建议：
A. 规则+AI混合（规则处理明确的，AI处理模糊的）
B. 纯AI分类（更智能，但需要训练数据）
C. 轻量方案（关键词+标签，简单快速）
D. 其他（请说明）
```

**结束时：** 能用一句话说清楚问题，用户确认方案方向。

---

### Stage 2: Design — 方案设计

**目标：** 搞清楚怎么做。

**执行方式：** 分块呈现方案，每完成一块等待用户确认。

**我会：**
- 提出设计方案，分 2-5 块呈现（按任务复杂度调整）
- 每块不超过 300 字
- 每块标注 [待确认]，等待你的反馈
- 用户确认后标记 [已确认]，再推进下一块
- 全部确认后，主动问"可以进入下一阶段吗"

**输出示例：**

```
## 方案块 1/3：整体架构

[待确认]
- 使用 Python + Flask 框架
- 项目结构：...
- 核心模块划分：...

## 方案块 2/3：数据模型
...
```

**结束时：** 所有方案块都标注 [已确认]，用户同意进入 Plan。

---

### Stage 3: Plan — 实施方案

**目标：** 搞清楚先做什么。

**我会输出：**
```
## 实施方案

1. [任务名]（复杂度：低/中/高）
   - 要做什么
   - 风险点

2. ...
```

**结束时：** 用户明确说 "go" 才进入执行。

---

### Stage 4: Use Case — 效果预览

**目标：** 用具体场景检验前三阶段的成果。

**定位：** Clarify + Design + Plan 三者的共同成果物。

**执行时机：** Plan 完成后、Execute 之前。

**我会：**
- 用具体场景描述"如果按这个方案做出来，会是什么样子"
- 包含具体数字和效果描述
- 像资深产品经理在陈述需求

**风格要求：**
- 口语化但不幼稚
- 具体场景 + 具体数字
- 不夸张，不煽情
- 给出"做完后的样子"——目录结构、文件命名、效果描述

**示例：**

```
📖 Use Case:

按这个方案做出来，你的使用场景：

场景：早上到公司，打开邮件
- 系统推送昨晚摘要：共 23 封，3 封重要
- 猎头来信，薪资涨 30%，已归档到 Work
- 快递到了，下午 2 点前取

场景：找邮件
- 输入关键词，0.5 秒找到
- 自动分类结果清晰可见

这是按我们的方案做出来的效果。
如果符合你的预期，说 "go" 开始执行。
如果不对，告诉我哪里需要调整。
```

**作用：**
- 让用户快速判断"这是不是我想要的"
- 减少理解偏差导致的返工

**结束时：** 用户说 "go" 进入 Execute。

---

### Stage 5: Execute — 执行

**前提：** 用户明确说 "go"

**我会：**
- 按计划逐步执行
- 每步完成后汇报
- 用 `todowrite` 跟踪进度
- 遇到问题停下来等你指示

---

### Stage 6: Review — 收尾

**我会：**
- 总结做完了什么
- 清理临时文件
- 提交有意义的 commit

---

### Stage 7: Compound — 经验输出

**定位：** Review 完成后，自动检测并输出经验。

**触发条件：** Review 阶段完成后，Agent 自动判断是否有值得输出的经验。

**检测标准：**
- 新的洞察或方法论
- 踩过的坑及解决方案
- 可复用的模式
- 与预期不符的学习
- 对未来工作有参考价值的内容

**触发示例：**
```
[Compound] 在这次任务中，我学到了一个洞察：
"使用 try-except 时，具体异常类型比宽泛的 Exception 更好"
要输出到 wiki 吗？

用户决策：
A. 写入 Wiki/concepts/   ← 概念/方法论类
B. 写入 Wiki/synthesis/ ← 综合分析/研究类
C. 写入当日日记  ← 临时记录
D. 丢弃              ← 不保存
```

**执行：**
- Agent 询问用户决策
- 用户选择后，Agent 执行写入操作
- 记录到知识库的对应位置

**本质：**
- 触发是 Agent 自动（减轻用户认知负担）
- 决策是用户控制（保持主导权）

**Human in Loop：** 在于"决策权"，不在于"触发权"

## 回退机制

**触发条件：** 当用户反馈与当前阶段成果不一致时。

（包括但不限于：用户说"不对"、新增需求、临时改想法、发现遗漏）

**判断方式：**

AI 分析反馈，判断问题在哪，建议退回阶段：

```
[分析] 你的反馈是关于方案方向，不是实施步骤。
[建议] 看起来应该退回到 Design 阶段，重新讨论方案。

可选：
1. 退回 Clarify（重新定义问题）
2. 退回 Design（重新设计方案）
3. 退回 Plan（调整实施步骤）
4. 重做 Use Case（调整预期描述）

你想退回到哪个阶段？
```

**原则：**
- AI 给出分析和建议
- 用户决定退到哪
- 不擅自跳转阶段

## Scaffold 管理

临时文件要有明确命名（如 `temp_*.py`），最终删除。

**收尾清单：**
- [ ] 确认生产文件
- [ ] 删除 `temp_*`、`debug_*`、`test_*` 等临时文件
- [ ] 有意义的 commit

## 边界说明

spec-first 解决 **"怎么做"** 的问题，不是 **"为什么做"** 或 "做什么"。

如果遇到方向困惑，这可能是另一个领域的问题（如价值观、自我认知），工作流解决不了。
