## Goal (1 sentence)

| Objective | Target | Measurement |
|-----------|--------|-------------|
| **accuracy** | 选项互斥、方案块覆盖关键决策点 | 每次使用后用户是否需要"退回"前一阶段 |
| **drift-resistance** | 用户需求在 7 阶段中不被扭曲或丢失 | 最终产出 vs Stage 1 问题的语义一致性 |

## How to run one session

> This section is for the **harness** (OpenClaw / Claude Code / Codex), not the human.
> After intake, the harness uses this runbook to execute sessions autonomously.

### Practice session (synthetic)

1. **Read**: Load this LOOP.md + the skill's SKILL.md
2. **Generate samples**: Create N synthetic test cases based on the skill's input domain (N=10 default; scale with objectives)
3. **Execute**: Run the skill on each sample (follow SKILL.md's defined workflow)
4. **Evaluate**: Score each output against training objectives:
   - accuracy: did the skill produce the correct result?
   - coverage: did the skill handle all expected patterns?
   - speed: token count / latency within budget?
   - robustness: did edge-case inputs break it?
   - drift-resistance: is the output consistent with the original goal?
   - ux-iteration: was the interaction too heavy / too light / formulaic / insufficient?
5. **Log**: Append results to `practice-log.md` (timestamp, sample count, scores, observations)
6. **Improve** (per decision_policy):
   - autonomous: AI proposes + applies improvements to skill config/rules
   - human gate: AI proposes, waits for approval
   - mixed: low-impact auto-applied, high-impact queued for review
7. **Check cadence**: If run count or time threshold hit → generate `evolution-report.md` + run `drift-check.md`

### Performance session (real)

Same as practice, but:
- Sample source = real workloads (not synthetic)
- Every change must be verified against real output (not just logged)
- Failures trigger recovery mode immediately
- **After session**: Save case snapshot to `cases/` directory (case-capture mechanism)

### Rest session (read-only)

1. Run skill on current input (real or synthetic)
2. Log observations to `practice-log.md`
3. Do NOT apply any changes
4. Flag anomalies for next practice/performance session

### Recovery session (post-failure)

1. **Rollback**: Undo the last applied change immediately
2. **Analyze**: Root-cause the failure (log in `practice-log.md`)
3. **Document**: Add failure pattern to `evolution-report.md`
4. **Wait**: Do NOT re-apply until root cause is documented and resolved

## Anti-patterns (this skill's specific)

- ❌ 走完 7 阶段但用户没真想清楚 — 检测：Stage 4 Use Case 时用户是否说"不对"
- ❌ 形式主义追问（问了但答案没用上）— 检测：追问答案是否出现在后续阶段
- ❌ 阶段跳跃不可追溯（用户不知道自己在哪）— 检测：每次阶段切换是否明确标注
- ❌ 过度结构化导致简单任务也走全流程 — 检测：简单任务的 Stage 1 是否可以快速 skip

## Drift signals

The loop is drifting from intent if:

- 用户频繁说"不用问，直接做"（说明流程太重）
- 最终产出与 Stage 1 问题语义偏离（drift-resistance 失败）
- 同一类任务反复触发回退（说明某个阶段设计有问题）
