# Case Study: 邮件自动分类系统

## 背景
用户 需要处理约 16000 封邮件，希望自动分类到不同文件夹。

## 实际工作流程（非 spec-first 的教训）

### 问题发现
1. **过早编码**：快速写了一个脚本，却没有先明确需求
2. **未考虑边界情况**：处理到一半才发现 Ollama 会超时、模型输出不稳定
3. **迭代返工**：多次重写分类逻辑和 prompt

### spec-first 应该怎么运作

#### Stage 1: Clarify
- **问题**：邮件分类的目标是什么？
- **回答**：清空收件箱，将邮件分类到 7 个文件夹
- **问题**：用什么模型？本地还是 API？
- **回答**：本地 Ollama（qwen2.5:3b）
- **问题**：分类体系是固定还是可以扩展？
- **回答**：先固定 7 类，发现不够再扩展

#### Stage 2: Design
**方案 A：一步到位**
```
收件箱 → Ollama 批量分类 → 移动
```
- 优点：简单
- 缺点：Ollama 单次处理有限制（50-75 封），会 timeout

**方案 B：分阶段**
```
Phase 1: 规则匹配 → 直接分类
Phase 2: 未匹配 → Other
Phase 3: Ollama 分类 Other
```
- 优点：可渐进式处理
- 缺点：需要维护状态

**选定**：方案 B

#### Stage 3: Implementation Plan
1. 实现 IMAP 连接和邮件读取
2. 实现规则匹配分类
3. 实现 Ollama API 调用
4. 实现分类结果解析
5. 实现邮件移动逻辑
6. 添加进度通知（Telegram）
7. 添加错误处理和重试

#### Stage 4: Execute
- 分阶段执行，每阶段验证
- 发现 Ollama 输出不稳定 → 调整 prompt
- 发现超时 → 减少批次大小

## 关键教训

### 1. 本地模型的不确定性
- **问题**：qwen2.5:3b 输出不稳定，偶尔返回 30+ 行而非 10 行
- **解决**：增加 `num_predict` 参数限制，正则解析只取有效分类

### 2. 中文编码问题
- **问题**：邮件标题包含乱码
- **解决**：使用 `latin-1` 作为 fallback 编码

### 3. 性能瓶颈分析
- Ollama API：85% 瓶颈
- IMAP 移动：15%
- **优化**：批次处理，减少 API 调用

### 4. 异步执行的必要性
- **问题**：长时间运行导致会话过大
- **解决**：后台运行脚本，定期发送进度

### 5. 脚手架未清理（系统臃肿根源）
- **问题**：开发过程中创建了 16+ 个临时脚本（test_*.py, debug_*.py, move_*.py 等），完成后未删除
- **影响**：系统变得臃肿，难以维护
- **教训**：spec-first 应包含"脚手架搭建和拆除"步骤
- **解决**：在 Review & Handoff 阶段明确删除临时脚本

## 最终配置

```python
BATCH_SIZE = 10
TIMEOUT = 120  # 秒
CATEGORIES = ['Promo', 'Work', 'Personal', 'Social', 'Receipt', 'Notify', 'Other']
```
