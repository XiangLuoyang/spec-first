# Spec-First: 系统性思维启动器

> 写代码之前，先把问题和方案想清楚。

**核心理念：**

不急着给答案，先搞清楚要解决什么问题。代码是最后一步，不是第一步。

---

## 触发方式

说清楚你的需求就行，例如：

- "帮我做个用户注册流程"
- "我想做一个邮件分类系统"
- "/think 怎么设计一个待办清单"

不需要说"用 spec-first"，我会自动判断并引导。

---

## 完整流程

### Stage 1: Clarify — 澄清问题

**目标：** 搞清楚到底要解决什么问题。

**执行方式：** 用 `question` 工具直接给出选项，不要先问开放性问题。

**我会：**
- 一次性给出 2-4 个互斥选项（用 `question` 工具）
- 选项要具体、可操作
- 每选项附简短说明

**示例（反例 vs 正例）：**

❌ 反例（先问开放问题）：
```
你希望达到什么效果？
请描述一下你的需求...
```

✅ 正例（直接给选项）：
```
你想达到什么效果？
A. 性能更好（响应更快、资源占用更低）
B. 更易维护（代码清晰、扩展性强）
C. 功能更强（新增 XX 功能）
D. 其他（请说明）
```

**结束时：** 能用一句话说清楚问题，用户确认。

---

### Stage 2: Design — 方案设计

**目标：** 搞清楚怎么做。

**执行方式：** 分块呈现方案，每完成一块等待用户确认。

**我会：**
- 提出设计方案，分 2-5 块呈现（按任务复杂度调整）
- 每块不超过 300 字
- 每块标注 [待确认]，等待你的反馈
- 用户确认后标记 [已确认]，再推进下一块
- 全部确认后，主动问"可以进入下一阶段吗"

**输出示例：**

```
## 方案块 1/3：整体架构

[待确认]
- 使用 Python + Flask 框架
- 项目结构：...
- 核心模块划分：...

## 方案块 2/3：数据模型
...
```

**结束时：** 所有方案块都标注 [已确认]，用户同意进入 Plan。

---

### Stage 3: Plan — 实施方案

**目标：** 搞清楚先做什么。

**我会输出：**
```
## 实施方案

1. [任务名]（复杂度：低/中/高）
   - 要做什么
   - 风险点

2. ...
```

**结束时：** 用户明确说 "go" 才进入执行。

---

### Stage 4: Use Case — 效果预览

**目标：** 用具体场景检验前三阶段的成果。

**定位：** Clarify + Design + Plan 三者的共同成果物。

**执行时机：** Plan 完成后、Execute 之前。

**我会：**
- 用具体场景描述"如果按这个方案做出来，会是什么样子"
- 包含具体数字和效果描述
- 像资深产品经理在陈述需求

**风格要求：**
- 口语化但不幼稚
- 具体场景 + 具体数字
- 不夸张，不煽情
- 给出"做完后的样子"——目录结构、文件命名、效果描述

**示例：**

```
📖 Use Case:

按这个方案做出来，你的使用场景：

场景：早上到公司，打开邮件
- 系统推送昨晚摘要：共 23 封，3 封重要
- 猎头来信，薪资涨 30%，已归档到 Work
- 快递到了，下午 2 点前取

场景：找邮件
- 输入关键词，0.5 秒找到
- 自动分类结果清晰可见

这是按我们的方案做出来的效果。
如果符合你的预期，说 "go" 开始执行。
如果不对，告诉我哪里需要调整。
```

**作用：**
- 让用户快速判断"这是不是我想要的"
- 减少理解偏差导致的返工

**结束时：** 用户说 "go" 进入 Execute。

---

### Stage 5: Execute — 执行

**前提：** 用户明确说 "go"

**我会：**
- 按计划逐步执行
- 每步完成后汇报
- 用 `todowrite` 跟踪进度
- 遇到问题停下来等你指示

---

### Stage 6: Review — 收尾

**我会：**
- 总结做完了什么
- 清理临时文件
- 提交有意义的 commit

---

## 回退机制

**触发条件：** 当用户反馈与当前阶段成果不一致时。

（包括但不限于：用户说"不对"、新增需求、临时改想法、发现遗漏）

**判断方式：**

AI 分析反馈，判断问题在哪，建议退回阶段：

```
[分析] 你的反馈是关于方案方向，不是实施步骤。
[建议] 看起来应该退回到 Design 阶段，重新讨论方案。

可选：
1. 退回 Clarify（重新定义问题）
2. 退回 Design（重新设计方案）
3. 退回 Plan（调整实施步骤）
4. 重做 Use Case（调整预期描述）

你想退回到哪个阶段？
```

**原则：**
- AI 给出分析和建议
- 用户决定退到哪
- 不擅自跳转阶段

---

## Scaffold 管理

临时文件要有明确命名（如 `temp_*.py`），最终删除。

**收尾清单：**
- [ ] 确认生产文件
- [ ] 删除 `temp_*`、`debug_*`、`test_*` 等临时文件
- [ ] 有意义的 commit

---

## 边界说明

spec-first 解决 **"怎么做"** 的问题，不是 **"为什么做"** 或 "做什么"。

如果遇到方向困惑，这可能是另一个领域的问题（如价值观、自我认知），工作流解决不了。
